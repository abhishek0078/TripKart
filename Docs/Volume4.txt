# 1. Purpose

This document defines the development rules for every AI coding assistant working on the TripKart project.

The purpose is to ensure that every generated file follows the same architecture, coding standards, naming conventions, and design principles.

No AI-generated code should violate these rules.

---

# 2. Core Philosophy

TripKart is **not** a Bus Booking App.

TripKart is **not** a Flight Booking App.

TripKart is a **Travel Platform**.

Bus, Flight, Train, Cab, Hotel and future services are Plugins.

The application is built around reusable engines.

Never duplicate features for individual plugins.

# 28. AI Prompt Template

Before generating code for any feature, the AI should assume the following:

* Respect the Plugin-Based Travel Engine architecture.
* Use SwiftUI and Swift 6.
* Follow MVVM + Clean Architecture.
* Use existing DesignSystem and UIFramework components before creating new ones.
* Never duplicate screens for Bus, Flight, or future plugins.
* Keep Views declarative and business logic inside ViewModels and Use Cases.
* All new code must be reusable, testable, and backend-ready.

---

# 29. Golden Rules

1. Reuse before creating.
2. Configure before subclassing.
3. Compose before duplicating.
4. Plugins provide data, not UI logic.
5. Views render state only.
6. ViewModels own presentation logic.
7. Repositories own data access.
8. Coordinators own navigation.
9. ThemeManager owns visual styling.
10. Every new feature must strengthenвЂ”not weakenвЂ”the architecture.

# 30. Conclusion

This document serves as the engineering contract for TripKart. Any developer or AI assistant contributing to the project must follow these rules to preserve consistency, maintainability, and scalability.

The goal is not simply to produce working code, but to build a travel platform that can evolve from Bus and Flight booking into a full ecosystem of travel services without architectural rewrites.
