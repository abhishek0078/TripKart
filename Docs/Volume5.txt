# TripKart

# Volume 5 – Data Architecture & API Contract

**Version:** 1.0

**Platform:** SwiftUI

**Architecture:** Plugin-Based Travel Engine

---

# 1. Purpose

This document defines how TripKart stores, manages, and exchanges data.

The goal is to ensure that the application can switch from local dummy JSON to REST APIs without changing the UI or business logic.

The Data Layer is responsible for:

* Local JSON
* Repository Pattern
* Data Mapping
* Future REST APIs
* Local Persistence
* Caching

---

# 2. Data Flow

Every feature follows the same flow.

```text
SwiftUI View

↓

ViewModel

↓

UseCase

↓

Repository

↓

Data Source

↓

JSON / API

↓

Repository

↓

ViewModel

↓

UI
```

Views never know whether data comes from JSON or APIs.

# 12. Search Contract

Every plugin implements the same search contract.

Input

* Origin
* Destination
* Travel Date
* Passenger Count

Optional

* Plugin-specific filters

Output

* List of Trip models

The UI never depends on Bus or Flight models directly.

---

# 13. Booking Contract

Booking Engine accepts

* Trip
* Travellers
* Contact Details
* Payment Method

Returns

* Booking

The plugin injects additional metadata if needed.

---

# 14. Ticket Contract

Ticket Engine accepts

Booking

Returns

Ticket

Plugins add only metadata.

No plugin creates its own ticket.

---

# 15. Payment Contract

Payment Engine accepts

* Booking
* Payment Method

Returns

* Payment Result

Future gateways

* Razorpay
* Stripe
* Cashfree

should implement a common PaymentProvider protocol.

Every endpoint should follow consistent rules.

Example resources

* /users
* /search

* /offers

Large collections should support

* page
* pageSize
* total
* hasMore

* Validation Error

---

# 26. Summary

TripKart's data architecture is built around stable domain models, repositories, and mappers. The UI communicates only with repositories and never with raw JSON or network responses.

By keeping repositories and domain models stable, the application can evolve from local dummy data to production APIs with minimal changes, preserving the Plugin-Based Travel Engine and ensuring long-term maintainability.
