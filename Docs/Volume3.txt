**Language:** Swift 6

**Minimum iOS:** 17+

**Architecture:** Plugin-Based Travel Engine

---

# 1. Design Philosophy

TripKart follows a **Design System First** approach.

Every screen must be assembled using reusable SwiftUI components.

No screen should build custom UI unless absolutely necessary.

Instead of thinking:

```
Bus Screen

Flight Screen
```

Think:

```
Reusable Components

+

Configuration

=

Screen
```

The Design System is the foundation of the application.

---

# 2. SwiftUI Principles

Every UI follows these rules.

### Declarative UI

Views describe UI.

They never control business logic.

---

### Composition Over Inheritance

Large screens are composed from many small reusable views.

Never create large 1000-line Views.

---

### Stateless Components

Small UI components should not own state.

State belongs inside ViewModels.

---

### Configuration Driven

Views receive models.

Not hardcoded values.

---

### Environment Driven

Theme

Localization

Session

Navigation

Settings

should come from Environment.

# 24. SwiftUI Best Practices

Use

* `@Observable` for view models (iOS 17+)
* `@Environment` for app-wide services
* `@Bindable` where appropriate
* `NavigationStack` for navigation
* `LazyVStack` and `LazyHGrid` for performance
* `ViewBuilder` to compose plugin-specific sections
* `AnyLayout` when layouts need to switch dynamically
* Custom `EnvironmentKey`s for theme, plugin context, and feature flags

Avoid

* Large monolithic views
* Business logic in Views
* Deeply nested `if` statements
* Inline styling
* Excessive use of `AnyView` (prefer generics and `ViewBuilder`)

---

# 25. Summary

The TripKart SwiftUI Design System is a **configuration-driven UI framework** built entirely around reusable components, view modifiers, styles, and design tokens.

The UI layer is independent of Bus, Flight, or future travel types. Plugins provide configuration models, while shared SwiftUI components render the interface. This ensures visual consistency, minimizes duplication, and makes it possible to add new travel services with little to no change to the UI layer.

This Design System acts as the foundation for every screen in TripKart and should be considered the only approved way to build user interfaces within the project.
